3. **숫자는 실상을 정직하게** — 회계가 손실을 과소표시하면 가장 위험.

### 1-B. 상시가동 호스트 (★사용자 결정·자원 필요★)
- **문제:** GitHub Actions = 해외 IP + 스케줄 드랍 + 6h 잡 한도. 폭락 정점에
  KIS API가 과부하로 전면 타임아웃돼 봇이 거래 불능이었다.
- **해법 옵션:**
  1. **소형 한국 VPS**(예: Naver Cloud / AWS ap-northeast-2 / Vultr Seoul)에서
     상시 `--loop` 실행 + systemd 자동재시작. 가장 안정적. 월 ~수천원~만원.
  2. 사용자 PC 상시가동 + 작업 스케줄러. 비용 0이나 PC 켜둬야 함.
  3. GitHub Actions 유지 + 재시도/워치독 강화(현 상태, 한계 있음).
- **추천:** ①. 결정·셋업은 사용자 몫 — 요청 시 배포 스크립트·systemd 유닛 제공.

## 2. 검증 프로세스 (라이브 즉석 수정 금지)

- **dry-run 스모크:** `mode=once --dry-run`으로 라이브 데이터 기반 "무엇을 할지"를
  주문 없이 확인. 모든 매매 변경은 이걸로 먼저 검증.
- **모의계좌 병행(있다면)** 또는 최소 1일 dry-run 후 라이브 반영.
- **회귀 테스트 필수** (현재 90+ 통과). 변경마다 추가.

## 3. 선제 포지셔닝 (폭락 *전에* 방어 전환)

- 레짐 전환(SMA200·카나리아)이 급락 속도를 못 따라간 게 6-08의 근본.
- 급락 조기 트리거(rapid_decline) 민감도 상향 + "하락 추세 확인 시 선제 인버스/방어"
  진입 조건 검토(단 휘둘림 리스크 → dry-run 검증 병행).

## 4. 공격적 수익 전략 (토대 검증 후)

- 레버리지 자율매매 dry-run 검증 → 수익 입증 후 실거래(사용자 선택).
- 인버스 능동 진입 튜닝.
